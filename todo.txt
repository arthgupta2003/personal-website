# [2026-03-09 22:30] TASK: Elo Taste Discovery System
# ============================================================
# Build a daily Elo ranking system that forces user to compare activity
# archetypes ("live jazz vs MIT lecture"), building a taste fingerprint
# that feeds into event recommendation scoring.
#
# Data model (db.py) — add two tables:
#   taste_items: (id, user_id, label, category, elo_rating DEFAULT 1500,
#                 matchups_count DEFAULT 0, created_at)
#     UNIQUE(user_id, label)
#   taste_matchups: (id, user_id, item_a_id, item_b_id, winner_id NULL=equal,
#                    created_at)
#
# DB methods:
#   - get_taste_items(user_id) -> list sorted by Elo desc
#   - get_next_matchup(user_id) -> pick two closest-Elo items (most informative)
#     Prioritize items with 0 matchups
#   - record_matchup(user_id, item_a_id, item_b_id, winner_id) -> Elo update K=32
#   - seed_taste_items(user_id) -> insert ~25 default archetypes
#
# Seed archetypes:
#   Live music (small venue), Concerts (big venue), Jazz/blues,
#   DJ/electronic night, Comedy show, Theater/improv, Art exhibition,
#   Film screening, Tech/AI talk, Science lecture, Book club/reading,
#   Board game night, Trivia night, Outdoor group fitness,
#   Climbing/bouldering, Yoga/meditation, Running club, Cooking class,
#   Pottery/maker class, Food pop-up/tasting, Wine/cocktail bar,
#   Coffee shop hangout, Networking event, Volunteer work, Dance class,
#   Walking tour, Museum visit, Hackathon
#
# Elo formula:
#   expected_w = 1 / (1 + 10^((loser_rating - winner_rating) / 400))
#   new_winner = winner_rating + K * (1 - expected_w)
#   new_loser = loser_rating + K * (0 - (1 - expected_w))
#   For "equal": use 0.5 instead of 1/0
#
# API endpoints (app.py):
#   GET  /api/taste/next           -> returns next matchup JSON
#   POST /api/taste/vote           -> {item_a_id, item_b_id, winner_id}, returns updated Elos
#   GET  /api/taste/rank           -> full taste stack JSON
#   GET  /taste                    -> web page: taste stack with Elo bars + inline vote widget
#   GET  /api/taste/vote-link?a=&b=&winner=&u= -> GET-based vote for email, shows confirmation
#
# Email (sender.py):
#   send_daily_matchup(email, token, item_a, item_b, dashboard_url, settings)
#   Subject: "Quick: Jazz night or climbing gym?"
#   Body: two items + three GET-link buttons [A] [B] [Equal]
#
# Ranking integration:
#   After computing event scores, map event category/vibe to closest taste_item.
#   Top-5 Elo items: +10 to event score. Bottom-5: -5.
#   Additive, not replacement.
#
# Scheduling:
#   scripts/send_daily_taste.py — cron daily 9am weekdays
#   Gets user, calls get_next_matchup, sends email.

# === Signal attribution ===
# For each scoring dimension, compute:
#   - Hit rate: % of events with high signal (>10) that were attended
#   - Miss rate: % of attended events that had low signal (<5)
#   - Lift: how much does a high signal increase attend probability vs baseline
#   - Example output:
#     "interest_score: 42% of events scoring >10 were attended (vs 8% baseline = 5.3x lift)"
#     "social_score: 35% lift, friend_score: 28% lift, discovery_score: 4% lift"
#   - This tells you which signals are actually doing work
#
# === Information Coefficient (IC) ===
# Rank correlation (Spearman) between each signal and user outcome:
#   - Outcome = attended (1/0), or rating (1-5), or RSVP status
#   - Compute IC per dimension per run, then average across runs
#   - IC > 0.1 = useful signal, IC < 0.02 = noise
#   - Track IC over time: is a signal getting better or worse as user
#     provides more feedback?
#   - Composite IC: correlation of final weighted score vs attendance
#
# === Decay analysis ===
# Do different signals predict better at different time horizons?
#   - urgency_score IC for events <3 days out vs >7 days out
#   - interest_score IC for events this week vs next month
#   - friend_score IC by time-to-event (friends going matters more last-minute)
#   - Hypothesis: urgency dominates short-term, interest dominates long-term
#   - Output: signal IC by days-until-event buckets (0-3, 3-7, 7-14, 14+)

#   - Simple approach: logistic regression where features = 7 dimension scores,

# ============================================================
# [2026-03-10] TASK: ML-based ranking models (beyond Claude-as-scorer)
# ============================================================
# Currently Claude is the entire scoring engine — it reads events and
# outputs 7 dimension scores. This works but has problems:
#   - Expensive ($3-4/run, scales linearly with event count)
#   - Non-deterministic (same event can score differently across batches)
#   - Slow (30+ API calls per run)
#   - No learning from user behavior (prompt is static, doesn't adapt)
#   - Claude is a generalist, not trained on YOUR preferences
#
# The move: use Claude for cold-start feature extraction, then train
# lightweight ML models on user behavior to do the actual scoring.
#
# === Phase 1: Feature engineering (Claude → features, ML → score) ===
# Keep Claude for extracting semantic features that are hard to compute:
#   - Event category/vibe classification
#   - Interest match (requires understanding user profile)
#   - Quality/reputation signals
# But use these as FEATURES into a learned model, not as final scores.
#
# Feature vector per event:
#   - Claude-extracted (cached, computed once per event):
#     category, vibe, interest_match_raw, quality_raw
#   - Computed deterministically (free, instant):
#     distance_km, day_of_week, hour_of_day, is_weekend, is_evening,
#     price_bucket, has_spotify_artist, source, venue_visit_count,
#     venue_avg_rating, taste_elo_match, friend_rsvp_count,
#     days_until_event, impressions_count, description_length,
#     attendee_count, is_recurring
#   - User behavior features:
#     category_attend_rate, source_attend_rate, venue_attend_rate,
#     avg_rating_for_category, time_slot_preference
#
# Model: gradient boosted trees (XGBoost or LightGBM)
#   - Target: attended (1/0) or rating (1-5) as regression
#   - Train on all historical (event_features, attended) pairs
#   - ~20 features, needs ~100+ attended events to be useful
#   - Retrain weekly after each pipeline run
#   - Feature importance gives you signal attribution for free
#
# Benefits over pure-Claude:
#   - Scoring is instant (no API calls) — <1ms per event
#   - Deterministic — same features = same score always
#   - Learns from YOUR behavior, not Claude's general knowledge
#   - Feature importance is interpretable
#   - Can score 10,000 events in <1 second
#
# === Phase 2: Hybrid pipeline ===
# Best of both worlds:
#   1. New events: Claude extracts semantic features (category, vibe,
#      interest_match) — cached per event, never re-computed
#   2. All events: ML model scores using full feature vector
#   3. Top-50 from ML: Claude does a final "narrative pass" —
#      generates match_reason explanations for the events you'll
#      actually show the user
#
# Cost: ~$0.50/run (Claude only for feature extraction on new events
# + narrative pass on top 50) vs $3-4 currently. 6-8x cheaper.
#
# === Phase 3: Embeddings + collaborative filtering ===
# Once you have multiple users:
#   - Embed events using sentence-transformers (title + description)
#   - User embedding = weighted average of attended event embeddings
#   - Score = cosine similarity between user embedding and event embedding
#   - Collaborative filtering: "users like you attended X"
#   - This is how Spotify/Netflix actually work at scale
#
# Implementation:
#   - sentence-transformers (all-MiniLM-L6-v2) for event embeddings
#   - Store embeddings in SQLite as BLOB (small scale) or faiss index
#   - User embedding updated after each attended event
#   - Cosine similarity as one more feature into the XGBoost model
#
# === Phase 4: Contextual bandits (explore/exploit) ===
# The discovery problem: how do you recommend things outside the user's
# known preferences without tanking precision?
#   - Thompson sampling: maintain uncertainty per category/vibe
#   - High uncertainty = more exploration (show it even if predicted low)
#   - After user feedback: update posterior, reduce uncertainty
#   - This is how you avoid the "filter bubble" problem formally
#   - LinUCB or neural contextual bandit on the feature vector
#
# === Implementation plan ===
# Dependencies: scikit-learn, xgboost (or lightgbm), sentence-transformers
# New files:
#   - src/recom/ranking/ml_model.py — feature extraction, model train/predict
#   - src/recom/ranking/embeddings.py — event/user embeddings
#   - scripts/train_model.py — offline training script
#
# Data requirements per phase:
#   Phase 1: ~100 attended events (a few months of usage)
#   Phase 2: same, just restructure the pipeline
#   Phase 3: ~200+ events and/or multiple users
#   Phase 4: ~500+ events with explicit feedback
#
# Dashboard integration:
#   - /admin/ml page: model performance metrics, feature importance chart,
#     Claude vs ML score comparison on recent events, training history
#   - A/B toggle: serve Claude scores vs ML scores vs hybrid to compare
#
# Start with Phase 1 (feature engineering + XGBoost) as soon as there are
# ~100 attended events in the DB. Until then, Claude scoring is fine —
# you need the training data first.
#
# === Training data bootstrap (don't wait for 100 real attended events) ===
#
# Strategy 1: Taste Elo as proxy labels
#   - Map each event to its closest taste archetype (by category/vibe)
#   - Use the Elo rating as a proxy preference score
#   - Event matches "live jazz" (Elo 1650) -> positive. "corporate networking"
#     (Elo 1200) -> negative. Instant labeled data for every historical event.
#   - Weak signal but covers the full event universe immediately
#
# Strategy 2: Event swipe game (Tinder for events)
#   - GET /taste/swipe — show real events from past runs, user swipes yes/no/maybe
#   - 10 swipes/day for a week = 70 labeled examples on REAL events
#   - Much richer than archetype Elo because it captures event-specific features
#     (venue, time, description, price) not just category
#   - Gamify: streak counter, "discover your taste profile" framing
#   - Can reuse events across runs — label once, train forever
#
# Strategy 3: Implicit signals from existing behavior
#   - Steering actions: "more" = positive, "not for me" = negative, "block" = strong neg
#   - Email clicks: which events user clicked vs scrolled past (add tracking
#     pixel + redirect links in email composer)
#   - Calendar feed: which events user kept vs deleted from their calendar
#   - RSVP data: going = strong positive, maybe = weak positive, cant = neutral
#
# Strategy 4: Claude distillation (immediate, zero user effort)
#   - Claude has already scored every event in the DB (7 dimensions + composite)
#   - Train XGBoost to replicate Claude's composite score from deterministic features
#   - The ML model learns Claude's scoring function but runs instantly for free
#   - Then fine-tune on real user behavior as it accumulates
#   - This alone eliminates $3-4/run cost with minimal quality loss
#   - Validation: hold out 20% of Claude-scored events, check R²
#
# Bootstrap priority order:
#   1. Claude distillation (zero effort, already have data, biggest cost savings)
#   2. Taste Elo mapping (zero effort, already have Elo data)
#   3. Event swipe game (needs UI, but high-quality labels)
#   4. Implicit signals (needs email tracking infra)

# [2026-03-10] TASK: Define north star metric + switch to categorical scoring
# ============================================================
# Problem: we don't have a clear objective function. "Score 0-100" is
# false precision — no meaningful difference between 67 and 71. And we
# can't evaluate the system without knowing what "good" means.
#
# === North star metric ===
# Primary: attend_rate × avg_rating
#   - "You went AND liked it" — the Sharpe ratio of recommendations
#   - High attend rate + low ratings = bad recs you went to out of boredom
#   - High ratings + low attend rate = good recs you never acted on
#   - Both components need to be high for the system to be working
#
# Secondary: discovery_rate
#   - % of attended events in categories you hadn't attended before
#   - Prevents filter bubble. The whole point is finding things you
#     wouldn't have found on your own.
#
# Health check: recommendation_coverage
#   - Are we showing diverse event types or converging to "jazz every week"?
#   - Entropy of recommended event categories should stay above threshold
#
# Track all three on /admin/metrics page. Plot over time.
#
# === Switch from 0-100 score to 1-5 categorical labels ===
#
# New scoring output from Claude (replaces 7-dimension 0-15 scores):
#   5 — Must go:     "Cancel other plans for this"    → auto-add to calendar
#   4 — Strong rec:  "Definitely consider this"       → show prominently
#   3 — Worth knowing: "If nothing better comes up"   → show in list
#   2 — Marginal:    "Not really for you"             → hide by default
#   1 — Skip:        "Wrong audience/interest"        → filter out
#
# Why this is better:
#   - Maps to real decisions (not arbitrary 0-100 scale)
#   - User rates attended events 1-5 → same scale, direct comparison
#   - Confusion matrix is trivial: predicted 5, actual 3 = clear overfit
#   - Claude is better at categorical judgment than precise numeric scoring
#   - Shorter responses = cheaper API calls
#   - Easier to explain: "we predicted 4, you rated 2 → we'll adjust"
#
# New Claude prompt (much simpler):
#   "Rate each event 1-5 for this user. Return:
#    event_id, rating (1-5), vibe (social/intellectual/mixed),
#    match_reason (one sentence)"
#   Keep vibe for weight/display purposes.
#   Optionally still ask for top 2-3 relevant dimensions as tags
#   (e.g. "interest, friend") for explainability, but not scored.
#
# Migration:
#   - Keep 7-dimension scores in DB for backward compat / analysis
#   - Add predicted_label (1-5) column to run_events
#   - New ranking prompt outputs label + reason (not 7 scores)
#   - _compute_score() becomes a simple lookup, not weighted sum
#   - Threshold changes: keep = label >= 3 (instead of score >= 25)
#   - Email/dashboard shows label as stars or tier name
#   - Backtester evaluates: predicted_label vs actual_rating
#     (confusion matrix, weighted kappa, mean absolute error)

#   Don't ask "is Claude worth $3/run" — ask "does Claude produce higher
#   attend_rate × avg_rating than a cheaper approach?"
#   Can't answer until we have data. Priority:
#   1. Use the product — attend stuff, rate it, generate training data
#   2. Build the backtest — measure which approach wins on the metric
